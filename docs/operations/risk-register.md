# Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| 동시에 여러 고객이 마지막 수량을 주문 | 초과 예약 | 서버 조건부 차감과 실패 응답 처리 |
| 운영자가 날짜별 메뉴 설정을 잊음 | 고객 예약 불가 또는 잘못된 예약 | 관리자 dashboard에 미설정 날짜 표시 |
| 픽업/마감 시간이 불명확함 | 문의 증가 | 날짜 카드와 예약 완료 화면에 반복 표시 |
| 사진 품질이 들쭉날쭉함 | 브랜드 신뢰 하락 | 실제 사진 우선, 밝기/비율 가이드 제공 |
| 관리자 화면이 복잡해짐 | 운영자 사용 포기 | MVP 기능 제한, 날짜/메뉴/예약 중심 유지 |
| 결제 미연동으로 노쇼 발생 | 매출 손실 | 초기에는 상태 관리와 안내 강화, 이후 결제 검토 |
| 개인정보 과수집 | 운영/법적 부담 | 이름/연락처/요청사항 중심 최소 수집 |
| Cloudflare D1 제약 | 쿼리/transaction 제한 | 단순 schema, 조건부 update, 서비스 계층 테스트 |
| 알림 미발송 | 예약 확인 누락 | MVP는 완료 화면 우선, 이후 알림 provider 추가 |

## Watch Items

- 예약 규모가 커지면 결제와 자동 알림 우선순위가 올라갑니다.
- 시즌 이벤트가 많아지면 메뉴 공개 예약, 기간 한정 메뉴, 할인 코드가 필요할 수 있습니다.
- 픽업 장소가 여러 개가 되면 `pickup_locations` 테이블이 필요합니다.
